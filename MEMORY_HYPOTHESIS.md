# Гипотеза: адаптивная память с метаслоем стратегий извлечения

## Проблема

Существующие системы памяти для LLM (RAG, MemGPT/Letta, векторные базы) имеют общий изъян: **стратегия извлечения знаний прошита в архитектуре**. Семантический поиск, граф, временная близость — выбирается один механизм и применяется ко всем данным одинаково.

Но разные типы знаний требуют разных стратегий:
- Нарративные данные (сценарий) → временно-причинное извлечение (что было до, кто участвовал)
- Персонажи → граф отношений (кто с кем связан)
- Лор/правила мира → иерархическое погружение (общее → частное)
- Факты и предпочтения → семантическая близость

Загрузка всего сценария в 200к токенов контекста не решает проблему — внимание модели размывается ("lost in the middle"). Нужна структурированная загрузка только релевантного.

## Ключевая идея

**Стратегия извлечения — это тоже данные, а не архитектура.**

Каждый узел знаний содержит не только содержимое, но и метаданные о том, как к нему обращаться:

```
name: season1/s01e01
description: Пилотный эпизод. Знакомство с Джоном и Мэри.
retrieval: narrative
related_load: characters/john, characters/mary
---
...содержимое эпизода...
```

```
name: characters/john
description: Джон Смит. Главный герой, детектив.
retrieval: graph
---
...характер, мотивы, арки...
```

```
name: series_lore
description: Правила вселенной, сеттинг.
retrieval: always
---
...лор...
```

### Типы стратегий (начальный список)

| Стратегия | Поведение |
|---|---|
| `always` | Всегда в контексте, не выгружается |
| `narrative` | При загрузке предлагает предыдущий/следующий узел |
| `graph` | При загрузке показывает связанные узлы в оглавлении |
| `semantic` | Поиск по смыслу через embeddings |
| `manual` | Только явная загрузка моделью (по умолчанию) |

## Иерархия как файловая система

Знания организованы в дерево через подпапки. Оглавление в системном промпте показывает только:
- корневые узлы
- детей уже активных узлов

Это имитирует человеческую память: вспомнил "сезон 1" → увидел список эпизодов → загрузил нужный → увидел персонажей этого эпизода.

```
personalities/
  series_lore.md          ← всегда виден
  characters/
    john.md
    mary.md
  season1/
    s01e01.md
    s01e02.md
  plot_threads.md
  timeline.md
```

## Калибровка через тестирование

Метаданные стратегии можно не только задавать вручную, но и **обучать**:

1. Пользователь или модель-тестировщик задаёт набор вопросов по базе знаний
2. Агент отвечает, пользователь оценивает качество
3. Система фиксирует: при каком наборе загруженных узлов ответ был правильным
4. Корректирует `retrieval`-параметры и `related_load` конкретных узлов

По сути это обучение гиперпараметров доступа к памяти, а не самой памяти. Аналог fine-tuning, но не весов модели, а структуры извлечения.

## Отличие от существующих подходов

| Система | Стратегия извлечения |
|---|---|
| RAG | Семантика, фиксирована |
| MemGPT/Letta | Семантика + временность, фиксирована |
| Generative Agents | Важность × давность × релевантность, фиксирована |
| **Эта идея** | Задаётся per-node, меняется моделью и пользователем |

Главное отличие: модель **сама осознанно** решает что загружать, опираясь на тип данных и контекст вопроса. Не автоматический поиск, а осознанная навигация.

## Текущее состояние реализации

`PersonalityProvider` (`src/memory/personality.py`) — плоская версия без иерархии и стратегий. Агент уже сам решает что загружать через `personality_load`. Это ядро идеи.

### Следующие шаги (по приоритету)

1. **Иерархия** — поддержка путей `characters/john` через подпапки
2. **Метаданные стратегии** — поле `retrieval:` в шапке файла
3. **`narrative` стратегия** — при загрузке эпизода предлагать соседние
4. **`always` стратегия** — узлы всегда в контексте без явной загрузки
5. **Калибровка** — режим тестирования и корректировки стратегий

## Связанные работы

- **MemGPT / Letta** (Packer et al., 2023) — LLM как ОС, управляет своей памятью
- **Generative Agents** (Park et al., Stanford, 2023) — importance × recency × relevance
- **A-MEM** (2025) — динамические связи между воспоминаниями
- **ACT-R** (Anderson, CMU) — когнитивная архитектура с activation-based retrieval
